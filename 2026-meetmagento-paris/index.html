Verification can't see this · every Magento dev knows tax rounding

Passes every gate. Still wrong.


public function ruleTotal(array $lines): float
{
    $total = 0.0;
    foreach ($lines as $line) {
        $price = $line['unit_price_cents'] / 100;            // money → float
        $total += round($price * $line['qty'] * 1.0825, 2);  // 8.25% tax, per line
    }
    return round($total, 2);
}
// phpstan: green · phpunit: green · lint: green · spec: met
// 3 lines: $64.92 ✓   ·   10k lines: off by $8.25   ·   100k: off by $82.50

↩ no machine flagged it, only validation against the contract does

This passes every gate — phpstan, phpunit, lint, the spec. And it's still wrong. It holds money in floats and rounds tax on every line — so the error compounds: nothing at three lines, eight dollars off at ten thousand, eighty-two fifty at a hundred thousand. You know this one: Magento literally lets you round tax per line or per total, and getting it wrong in custom pricing code drifts the ledger. No machine flagged it. The class of bug you prevent, not re-find

Encode the rule once


# deptrac.yaml — enforce Magento's layered architecture
ruleset:
  Presentation:    [ServiceContract]  # Controller / Block / ViewModel
  ServiceContract: [Model]            # Api/ interfaces + repositories
  Model:           [Persistence]      # domain → Model/ResourceModel
  Persistence:     ~                  # the DB layer is a leaf
          

$ vendor/bin/deptrac analyse
  Vendor\Promo\Block\Cart\Discount must not depend on Vendor\Promo\Model\ResourceModel\Rule
  reason: a Block may only reach a ServiceContract — inject RuleRepository
  Violations  1            # red, before a human ever opens the PR
You encode the rule once, in Magento's own grain. The presentation layer — controllers, blocks, view models — may only reach a service contract, the Api interfaces. It must never touch a ResourceModel directly. Run Deptrac, and a block that queries the database goes red before a human ever opens the PR. You stop re-finding the same layering breaks and N+1s — you prevent them. — backstage — The violation is the classic Magento smell: a Block reaching straight into a ResourceModel/collection instead of injecting the repository (service contract). Real output pattern from the demo repo (branch demo/deptrac-violation, kept off main). Complementary: bitexpert/phpstan-magento ships rules for the ObjectManager / in-loop anti-patterns; Deptrac handles the layering.

Context is more than code

Four things the agent needs to know

Intent

The Bet, the constraints, the invariants, what "done" means and what must never break.

Environment

The live codebase, dependencies, tests, runtime, the system as it actually is.

History

PRs, commits, and the why behind past decisions, ADRs, not just the diff.

People

Contributors and ownership, who understands which system, and the scars they carry.

The agent needs four things: intent, environment, history, and people. Code is the smallest part. Intent, history, and the people who carry the scars — that's what agents and juniors usually lack.

Stop writing requirements. Start writing Bets.

A requirement prescribes a solution. A Bet states what you believe, how you'll know, and what you won't risk.


# Bet · Cart price rule
Appetite: 2 weeks · One-way door: no

## Hypothesis
A "buy 3, 10% off" automatic rule lifts units/order.

## Signal (validates)
Units per order — not discount $ given.

## Guardrail
Exclusive with coupons; never re-discounts reduced
items; reconciles in credit memos.
Stop writing requirements. Start writing Bets. A requirement prescribes a solution. A Bet states what you believe, how you'll know, and what you won't risk — an appetite, a signal, and a guardrail. Note the signal is units per order, not dollars discounted — that's exactly what the e2e validates against. — backstage — Appetite + the one-way-door flag come from Shape Up (Ryan Singer); the record form is Nygard's ADR (2011). This is the cart-price-rule Bet the Grilling Session and the demo both resolve.
