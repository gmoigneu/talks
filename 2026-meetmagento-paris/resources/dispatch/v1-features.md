# V1 — features

Companion to [v1-alignment.md](v1-alignment.md). Translates the V1 scope into a list of capabilities the product must deliver by end of June 2026. Each feature has a one-paragraph description, the acceptance signal that proves it works, and pointers to the screens that deliver it.

Pairs with [v1-screens.md](v1-screens.md). Strict V1 throughout — anything labelled **[STRETCH]** is something the discovery feedback strongly suggests we ship but is not in the v1-alignment baseline; it requires an explicit scope decision before engineering picks it up. Anything labelled **[POST-V1]** is in [future-scope.md](future-scope.md) and is not designed here.

---

## 1. Onboarding and setup

**F-01 · Signup.** New user creates a Dispatch account. Authentication runs through Upsun Auth (`auth.upsun.com`), the cross-product Upsun identity service — the same identity layer used across the Upsun product family. Users do **not** need to be Upsun Cloud customers. If they already have an Upsun identity (because they use any Upsun-family product), they sign in with it. If they don't, one is created during signup. The available sign-in methods are whatever Upsun Auth exposes — GitHub OAuth, Google OAuth, email/password, etc. — and are not Dispatch-controlled. Auto-bootstrap then creates `<user>'s Org` + a workspace called `Default` + an `Admin` team, with the new user holding `owner` role on the org membership and seated on the Admin team. No UI to pick names or create additional Orgs/Workspaces/Teams at V1. The new account lands on an empty workspace dashboard. *Acceptance:* a new user can sign up via Upsun Auth and reach the empty workspace dashboard in under 90 seconds; the `identities`, `organizations`, `workspaces`, `memberships`, `teams`, `team_members`, and `team_workspaces` rows all exist after first sign-in. *Screens:* S-01, S-02, S-03.

**F-02 · GitHub App install and disconnect.** User clicks "Connect GitHub" and is taken through the GitHub App install flow at the GitHub Organization scope; selected repos are made accessible in the Workspace. Owners and admins can later disconnect the GitHub App from the GitHub connection settings (S-19). Disconnect is a destructive, audited action: all in-flight runs are interrupted, every installed workflow on every affected repo is automatically paused, and a workspace-wide indicator appears explaining that GitHub is disconnected and what the user needs to do to recover. Historical run data is preserved — disconnect does not delete the audit trail or the previous-runs table. *Acceptance:* user can pick a repo set on GitHub's side and return to Dispatch with those repos listed; disconnecting interrupts every in-flight run cleanly, pauses every workflow, surfaces the disconnect state on the workspace overview, and leaves historical data intact. *Screens:* S-04, S-05, S-19, S-10.

**F-03 · BYO key configuration.** User adds an Anthropic and/or OpenAI API key at the Org level. One key per provider. V1 does not support per-Workspace overrides. On save, Dispatch issues a real test call against the provider's API; on success the validation pill shows green with the model name the test call returned and the flow advances to the next setup step; on failure an inline error explains what failed (invalid key, rate limit, network) and the key is not stored. *Acceptance:* a user-supplied key is stored encrypted only after a successful test call, validation feedback is shown, and at least one valid provider is required before a workflow can run. *Screens:* S-17, S-18. 

**F-04 · Workflow catalog.** The catalog is the browseable library of workflow types available in the workspace — PR Review and Issue to PR in V1, growing as the templates library lands post-V1. Users discover and preview workflow types here, then click Install on a card to launch the install flow (F-05). Reachable from the sidebar at any time, from S-03 first-time setup, and from S-09 Workflows list "Add new." *Acceptance:* a user can browse the catalog, read every workflow type's description, diagram, trigger summary, and gate summary, and launch the install flow from any card. *Screens:* S-06.

**F-05 · Workflow install.** Install a single **workflow instance** — one configuration plus one set of repos. Each pass through the install flow creates exactly one instance; all repos in that instance share the same settings. To run the same workflow with different settings on different repos, the user creates a second instance. A workspace holds many instances of the same workflow type, each scoped to its own repos and config. **Uniqueness rule:** a given repo can belong to at most one instance per workflow type. The same repo may appear in instances of *different* workflow types simultaneously (for example, a PR Review instance and an Issue to PR instance on the same repo are fine), but two instances of the same type cannot share a repo — the runtime needs an unambiguous configuration to apply when a trigger fires on that repo. Instances can be **paused** (webhook firing stops, history kept) and **resumed** without re-install. **Uninstalling** an instance stops it firing on future triggers; historical traces (runs, decisions, costs) stay queryable from the previous-runs table (F-15) and the audit log (F-20). *Acceptance:* a user launches the install flow from F-04 with a workflow type pre-selected, sets its configuration, attaches a set of repos, and creates the instance; the new instance appears in S-09 Workflows list and is bound to the configured repos; the install is rejected if any selected repo already belongs to another instance of the same workflow type. *Screens:* S-07 (install), S-09 (Workflows list), S-15 (Workflow instance overview).

**F-06 · Team invitation.** Two-step flow: (1) An owner or admin in the Org creates an invitation for an email address. The invitee receives an email with the accept link. (2) On clicking the link, the invitee signs in via Upsun Auth (per F-01) if they aren't already authenticated. Invitations expire after 7 days; the owner or admin can cancel or re-send a pending invitation from S-20. *Acceptance:* an invited teammate can complete signup and immediately see the same workspace dashboard the owner sees, with approval permission on the workflows; expired or cancelled invitations are no longer redeemable. *Screens:* S-19, S-20.

## 2. Workflows

**F-07 · PR Review workflow (webhook-triggered).** A connected repo gets a PR opened. The PR Review workflow is triggered automatically. The agent gathers PR context (diff + linked issue + reviewer history), runs spec-compliance + architecture + breaking-change checks, and posts a structured review as a PR comment. No approval gate. The review is retriggered on each new commit until either the per-PR review limit is reached or the PR is merged. Once the limit is reached, additional reviews can either be triggered manually (per F-09) or the limit can be raised or removed in workflow settings. PR Review does **not** fire on draft PRs unless manually triggered; when a closed PR is reopened the workflow re-runs against the current state with the per-PR review-limit budget reset. Force-pushes that don't change file content (pure rebase) terminate with status `skipped` and no new review comment. Dispatch enforces a per-Org concurrency cap (2 concurrent runs by default per [v1-architecture.md §4.3](../engineering/architecture/v1-architecture.md#43-concurrency-and-rate-limits)); additional triggers queue with status `queued` and run FIFO as runs complete. *Acceptance:* opening a PR on a connected repo produces a Dispatch-authored structured review comment on that PR within the agreed SLA. *Screens:* S-08, S-11, S-12.

**F-08 · Issue to PR workflow (user-triggered).** A user mentions `@upsun-dispatch` on a GitHub issue **or assigns `upsun-dispatch` as the issue assignee** (per v1-alignment.md §2.4). The workflow gathers context, runs a planning agent, posts the plan as a comment on the issue, pauses for plan approval. On plan approval, the coding agent implements and a critic agent reviews. A draft PR opens. The Issue to PR flow ends here — full stop. If PR Review is installed on the repo, it runs separately on the new PR (per F-07's own trigger semantics, including its draft-PR rule); otherwise the PR sits open for the human to handle. Dispatch is never the GitHub-side **author** of a PR — only a **co-author**. The author is the issue's assignee (the human who took ownership of the issue); if the issue was assigned to Dispatch directly, the author is whoever made that assignment. When the draft PR opens, Dispatch also posts a comment on the originating issue linking to the new PR. Issue to PR does not auto-merge or auto-close the originating issue in V1 — merge stays a human action on GitHub. *Acceptance:* end-to-end Issue → Plan → Plan-approval → Implementation → Critic → Draft PR opens, against a real repo, with the plan approval gate working and the originating issue carrying a Dispatch-authored comment pointing at the new PR. *Screens:* S-08, S-11, S-12, S-13, S-14.

**F-09 · Manual trigger.** Either workflow can also be triggered manually from the workspace dashboard for an arbitrary repo/issue/PR, through a user mention of `@upsun-dispatch` in the right place and format, or for Issue to PR specifically by assigning `upsun-dispatch` as the issue assignee. *Acceptance:* a user can click "Run" from the dashboard and select a target on demand, leave a comment on the PR or issue, or assign Dispatch as the assignee. *Screens:* S-08 (with a "Run" button).

## 3. Approvals and decisions

**F-10 · Plan approval gate (Issue to PR only).** When the planning agent posts a plan on the issue, the workflow pauses. Approvers reply on the issue with `approve`, `reject`, or `revise: <feedback>`. Two approver populations are accepted (per [identity-and-auth.md §10](../engineering/auth-permissions/identity-and-auth.md#10-first-touch-github-commenter-policy)): (a) Dispatch Team members in the workspace whose linked `identities.github_login` has GitHub repo write+ access; (b) **first-touch GitHub commenters** — GitHub users with repo write+ access who have never signed in to Dispatch and have no `identities` or `memberships` row. The second population is ambient access via GitHub, not Dispatch onboarding. On `revise`, the planning agent iterates and posts an updated plan. The loop continues until approval, rejection, or max iterations is hit (max iterations = 5 per [v1-architecture.md §6.2](../engineering/architecture/v1-architecture.md#62-sprints-2-3-issue-to-pr-engine-proof)). The approval loop also has a 7-day total budget across iterations; on expiry the run terminates with `approval_timeout`. The plan is also surfaced inside Dispatch with a "View plan in Dispatch" link from the issue comment; that link requires authentication, so first-touch commenters interact via GitHub only. This is a read link only — there is no "Approve in Dispatch" action; the approval, rejection or demand for revision happens directly on GitHub. Dispatch tracks it all. Approval rights are gated by having a linked GitHub identity with repo `write+` access — a Dispatch user who signed up via a non-GitHub method (Google, email/password) cannot approve plans until they connect their GitHub identity from their user profile (per F-26). When a recognised approval comment (`approve` / `reject` / `revise:`) lands on the GitHub issue, Dispatch acknowledges it immediately with a reaction on that comment (👍 / ❌ / 🔁) so the human knows the comment was parsed, and the decision is recorded in the run's decisions section (S-11) within a couple of seconds — no manual refresh required. *Acceptance:* a plan can be approved (continues), rejected (run ends), or revised iteratively (loops) from both GitHub and Dispatch surfaces; an unlinked Dispatch user is blocked from approving with a clear Connect-GitHub prompt; every approve/reject/revise comment gets a reaction acknowledgement on the GitHub issue within seconds; decision rows appear on the run detail page in real time. *Screens:* S-13.

**F-11 · Approvals queue (centralized).** Workspace dashboard surfaces all currently-pending **plan approvals** across all Issue to PR workflow runs in one view. Each item shows: workflow, run, repo, time-pending, and a one-click action that opens the plan approval surface (S-13). A count badge in the workspace navigation indicates how many plan approvals are waiting on the current user. *Acceptance:* an approver opening Dispatch sees every pending plan approval they're authorized to make in a single list, with the nav badge matching the count. *Screens:* S-08, S-14.

## 4. Run observability

**F-12 · Workspace overview.** The workspace dashboard is the home of Dispatch — the screen the user opens daily to see what's happening in their workspace. It surfaces, in a single scroll: any runs currently in flight (F-14), pending plan approvals waiting on the current user (F-11), and the previous-runs table covering all workflows in the workspace (F-15). It's also the entry point to every other surface — individual run detail (F-13), workflow overview (F-16), settings (F-24), and the user's own profile (F-26). *Acceptance:* an authenticated user lands on the workspace overview by default after sign-in; every active concern (live runs, decisions to make, recent activity) is visible without further navigation. *Screens:* S-08.

**F-13 · Run detail page.** Every run gets a dedicated page. Contains: trigger metadata, the context bundle the agent saw, the agent's reasoning trace, review findings, diff (where applicable), every approval decision with actor/timestamp/diff-hash, and the running cost. Each decision row deep-links to the specific GitHub comment that triggered it. The whole trigger-to-result chain is visible on one page per v1-alignment §2.2. For PR Review re-runs after new commits, the page also surfaces the **review scope** — full vs. incremental (with the focus file list) vs. skipped (with the reason, e.g. pure rebase) — so a reviewer watching a commit-heavy PR can see exactly what Dispatch reviewed this time. A **Cancel** action terminates in-flight runs cleanly (with an audit entry); a **Re-run** action on completed, failed, or cancelled runs. For **failed** runs within 24 hours of the failure, Re-run resumes from the failed step (preserving prior context, partial output, and counters). For every other case (completed runs, cancelled runs, and failed runs older than 24 hours), Re-run starts a new run from scratch against the same trigger source (same PR or issue) with the current workflow configuration — the new run gets fresh review-limit and budget counters. Status pills use a defined vocabulary derived from [v1-architecture.md §5](../engineering/architecture/v1-architecture.md#5-recovery-and-failure-cases): `queued`, `running`, `completed`, `failed`, `skipped`, `cancelled`, `budget_exhausted`, `secret_denied`, `max_iterations`, `approval_timeout`, `rejected`. *Acceptance:* opening any run produces a single page that explains what happened, why, who decided, and what it cost; Cancel and Re-run work from this page; status pills match the engineering vocabulary. *Screens:* S-11.

**F-14 · In-flight run view.** While a run is executing, the run detail page updates in real time. User can watch reasoning, see the live cost ticker, see the current step, and stream live logs from the execution sandbox (per [sprint-1-execution-plane-plan.md W5.3](../engineering/sprints/archive/sprint-1-execution-plane-plan.md#w5-3) — 64 KiB / 500 ms chunked relay). *Acceptance:* a watcher sees state changes within seconds of them happening server-side, including new log lines. *Screens:* S-11 (live mode).

**F-15 · Previous runs table.** Workspace dashboard shows previous runs across all workflows with status, workflow, repo, trigger, duration, cost, and a link to the run detail. Supports filtering by workflow, repo, status, and date range, plus free-text search by PR or issue number. *Acceptance:* the user can scan the last 20+ runs from the dashboard without leaving the page, and get to the older ones if needed; filters and search return matching runs. *Screens:* S-08.

**F-16 · Workflow instance overview.** Each workflow instance (per F-05) has a dedicated page where owners and admins manage that instance's behaviour. The page shows: a static diagram of the workflow's steps and gates (the structure is read-only in V1 — no visual builder yet, this is the seed for one), the list of repos in this instance, the runs from this instance's history, and a **Settings** panel for the instance's configuration (PR Review per-PR review limit and manual-retry toggle, plan-gate maximum iterations for Issue to PR), plus lifecycle actions (Pause, Resume, Uninstall per F-05). The schema comes from the workflow definition; the values are set on the instance and apply uniformly to every repo in the instance. If the user wants different values on different repos, they create a second instance (back to F-04 catalog and F-05 install). Settings changes apply to new triggers only; in-flight runs keep the configuration they started with. *Acceptance:* clicking an instance from S-09 opens its page with the diagram, the repo list, recent runs for that instance, and the settings panel; saving a settings change applies it to subsequent runs only. *Screens:* S-15 (with S-09 as the nav parent that lists every workflow instance in the workspace).

## 5. Cost and spend control

**F-17 · Per-run cost.** Every run records token usage, model, time, and total dollar/euro cost. Surfaced on the run detail page and in the recent-runs table. *Acceptance:* every completed run carries an accurate per-run cost figure based on real provider invoicing math. *Screens:* S-08, S-11.

**F-18 · Per-workflow cost trends with outlier flagging.** The workspace dashboard shows aggregate cost trends per workflow over time (last 7/30 days). Outlier runs (defined as >2× the workflow's recent median) are flagged with a visual marker. *Acceptance:* outlier runs are visibly distinguishable on the dashboard without requiring the user to dig into the run page. *Screens:* S-08. **[STRETCH for V1 depending on engineering capacity — the underlying JSONL capture is in V1, the visualization may slip.]**

**F-19 · Org-level spend caps, alerts, and quotas.** Owner can set a hard monetary cap at the Org level. Two gates protect the cap: (a) pre-run gate that blocks new runs that would exceed the cap; (b) pre-step gate that blocks mid-run before the next step that would exceed the cap starts. The agent fails the run cleanly with an audit entry. Below the hard cap, Dispatch surfaces **alert thresholds at 50%, 75%, and 90%** of the period's spend (per v1-alignment.md §4.2) as a banner on the workspace overview and as an audit-log row — gives owners and admins a heads-up before runs actually start blocking. The workspace overview also surfaces engineering's **hard quota envelope** (per [v1-architecture.md §4.3](../engineering/architecture/v1-architecture.md#43-concurrency-and-rate-limits)) — daily run cap, wall-clock cap per run, and agent-loop cap per run — each shown as current usage vs. limit so customers can see when they're approaching a constraint before runs queue or fail. *Acceptance:* a configured monetary cap stops runs at the right point; the 50/75/90% alert thresholds are visible without requiring email notifications; quota usage is visible on the workspace overview and surfaced on the affected run's detail page. *Screens:* S-08, S-17, S-21.

## 6. Audit and identity

**F-20 · Decision audit log.** Every plan-stage decision (plan approval, plan rejection, plan revise) is recorded with actor, timestamp, decision type, the plan content at decision time, and a link back to the originating run. Cap-block events from F-19 are also recorded. Surfaced on the run detail page in a dedicated section, and queryable across runs via the audit log view. PR-stage merge decisions are out of scope in V1 (the team approval gate is deferred to [future-scope.md](future-scope.md)). Audit rows are retained for 1 year per [v1-architecture.md §4.4](../engineering/architecture/v1-architecture.md#44-audit-and-retention). The audit-log view (S-16) also surfaces and filters **Upsun staff access** rows for any rare break-glass / support actions on the workspace, so customers can audit who from Upsun touched their data. *Acceptance:* given any run, the full sequence of plan-stage human decisions and cap blocks can be reconstructed; audit rows older than 1 year are no longer queryable; staff-bypass actions appear in the audit log with a distinct marker. *Screens:* S-11, S-16.

**F-21 · Identity-separated agent.** Dispatch's agent acts under its own GitHub identity (`upsun-dispatch`), distinct from the operator's. Agentic GitHub actions (commits, PR opens, comments) are marked as co-authored between `upsun-dispatch` and the human who triggered the workflow. Every Dispatch-authored commit and PR description also carries a `Dispatch-Run: <run_id>` trailer (per [v1-architecture.md §4.4](../engineering/architecture/v1-architecture.md#44-audit-and-retention)) so anyone reading the Git history can navigate from a commit or PR back to the originating Dispatch run. *Acceptance:* a GitHub viewer can distinguish what Dispatch did from what a human did, the co-authorship trail is intact, and the Dispatch-Run trailer on commits and PR descriptions links back to the corresponding run page. *Screens:* visible on S-11 in the actor field. No dedicated UI.

**F-22 · JSONL telemetry capture (internal).** Every interaction, prompt, response, decision, and lifecycle event is captured as JSONL from day one. **Engineering-internal — never exposed to customers.** It's the foundation for the future autonomy dial, the continuous-improvement loop, and the future billing ledger. Customer-facing traceability is delivered by F-13 (run detail page) and F-20 (decision audit log); JSONL itself is not. *Acceptance:* every run produces a complete JSONL trace persisted internally, queryable by engineering. *Screens:* none.

**F-23 · First-touch GitHub commenter policy.** A GitHub user with `write+` access on a connected repo can trigger Dispatch workflows by commenting `@upsun-dispatch` on an issue or PR, and can approve plans on the issue, **without ever signing in to Dispatch and without any `identities` / `memberships` / `team_members` rows existing for them**. This is intentional ambient access per [identity-and-auth.md §10](../engineering/auth-permissions/identity-and-auth.md#10-first-touch-github-commenter-policy), not onboarding — they have no Org, no Workspace, no Team, no Dispatch UI surface. Their actions appear in audit logs rendered as `<github_login> (GitHub)` (per [identity-and-auth.md §7](../engineering/auth-permissions/identity-and-auth.md#7-permission-derivation)) — for example "@jane (GitHub)" — so anyone auditing the log can distinguish a registered Dispatch teammate from an external GitHub-only actor at a glance. This policy is what lets contractors, freelancers, and external reviewers participate in plan-stage decisions without a separate Dispatch invite cycle. *Acceptance:* a GitHub user with repo write+ access who has no Dispatch account can trigger `@upsun-dispatch` on an issue and approve the resulting plan on the issue, with each action recorded in the audit log as `<github_login> (GitHub)` and visually distinct from registered Dispatch users. *Screens:* none — by design, no Dispatch UI surface for this population.

## 7. Settings and administration

**F-24 · Org settings page.** Single page where owners and admins manage the Org: BYO keys (F-03), GitHub installation and the connected-repos list (add or remove repos), spend caps (F-19), and team membership and invitations (F-06). Workflow-specific configuration — review limits, manual-retry toggles, plan-gate iteration counts — lives inside each workflow instance itself (F-05, F-16, S-15), not here. Members have read-only access to settings; admins can change most things; owners can do everything, except transferring ownership of the Org (Org ownership transfer is [POST-V1] — see [future-scope.md](future-scope.md)). *Acceptance:* every Org-level administrative setting Dispatch supports in V1 is reachable from this one page, gated by role. *Screens:* S-17, S-18, S-19, S-20, S-21.

**F-25 · Empty states.** Every "list" surface (workspace dashboard, recent runs, approvals queue, audit log) has a designed empty state that explains the next action. *Acceptance:* a brand-new account has no zero-state "blank page" experience. *Screens:* baked into S-08, S-14, S-16.

**F-26 · User profile and sign-out.** Each user has a profile page where they can see their account details, manage how they sign in, and sign out of Dispatch. The important capability here is **connecting a GitHub account** for users who signed up another way (Google, email/password). Plan approval (F-10) is gated by whether the approver's GitHub identity has write access to the repo — so if someone joined Dispatch via Google and now needs to approve plans, they add the GitHub link from their profile. The top-bar avatar gives a quick path to the profile and a one-click sign-out from any screen. *Acceptance:* a user can sign out from any screen and is returned to the sign-in page; a user who signed up without GitHub can later connect their GitHub account from the profile page and immediately gain plan-approval rights on repos where that GitHub account has write access. *Screens:* top bar (avatar + sign-out), S-22.

- CLI surface — v1-alignment §2.2 says "The CLI will follow up right after." Out of scope for end of June.
- Custom or user-authored workflows — explicitly **[POST-V1]**.
- Visual workflow builder — explicitly **[POST-V1]**; F-16 is the read-only seed.
- Template editing / Templates Library — explicitly **[POST-V1]**.
- Multi-stage approval gates with role routing — explicitly **[POST-V1]**.
- Autonomy dial in the UI — explicitly **[POST-V1]**; data model and JSONL capture (F-22) are in V1 so the future dial has a corpus.
